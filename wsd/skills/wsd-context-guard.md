# WSD 上下文守卫技能

## When to Use

在以下情况激活：
- `wsd:execute` 执行多个任务时
- 长时间工作会话（超过1小时）
- 用户反映"感觉AI开始出错了"或"回答质量下降了"
- 准备启动新的子代理时

## How It Works

### 上下文使用量估算

根据当前对话轮次和内容量估算上下文使用率。

### 三级响应策略

**绿灯（< 60%）**：正常执行，无特殊操作

**黄灯（60-80%）**：发出警告

```
⚠️ 上下文警告（约 65% 已使用）

当前会话的上下文已使用较多，可能影响后续任务质量。

建议：
  • 当前任务（T-005）完成后，开启新会话继续
  • 在新会话中运行：/wsd:execute REQ-20260414-001 T-006
  • WSD状态会自动从 .wsd/ 目录恢复

继续当前任务？(yes/no):
```

**红灯（> 80%）**：阻止新任务

```
🛑 上下文已满（约 85% 已使用）

为保证代码质量，不建议在此上下文中继续执行新任务。

当前状态已保存：
  进度：4/6 任务完成
  下一任务：T-005（编写单元测试）

请开启新会话，然后运行：
  /wsd:execute REQ-20260414-001

WSD 会自动从保存的进度继续。
```

### 子代理自动隔离

`wsd:execute` 为每个任务创建独立子代理，天然避免 Context Rot：
- 子代理获得完整但精确的任务上下文包
- 子代理完成后销毁，不污染主会话
- 主会话只维护元状态（哪个任务完成了）

### 上下文包构建原则

为子代理构建上下文包时：
- **只包含该任务需要的规格**（不是整个规格库）
- **只读取相关代码文件**（不是整个代码库）
- **包含完成标准**（明确任务边界）
- **包含已完成任务的 commit SHA**（不是代码内容）

## Examples

### 示例 1：正常执行流

```
任务 T-001 → 子代理执行 → 完成 → 销毁 → 主会话记录结果
任务 T-002 → 新子代理 → 执行 → 完成 → 销毁 → 主会话记录结果
...
```

### 示例 2：手动触发

```
用户：这个会话感觉有点乱了，帮我检查一下

上下文守卫：
当前上下文估计使用量：约 72%

建议在完成当前任务后切换会话。
当前进度（T-003 执行中）将自动保存。
```
