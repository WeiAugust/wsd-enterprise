# WSD 规格编写指南

## When to Use

- 用户在编写 functional.md 或 acceptance.md
- 用户审查 wsd-analyst 生成的规格文档
- 规格质量评审时
- 新人学习规格编写方法时

## How It Works

### BDD 规格黄金法则

**好的规格**特征：
1. **用户语言**：用业务术语，不用技术术语
2. **行为描述**：描述"做什么"，不描述"怎么做"
3. **可测试**：可以直接转化为自动化测试
4. **独立性**：每个场景可以独立理解和测试

**反模式**（避免）：
- ❌ "调用 POST /api/v1/users 接口" → 这是实现细节
- ✅ "用户提交注册表单"
- ❌ "数据库中插入一条记录" → 这是内部实现
- ✅ "系统创建用户账号，用户可以立即登录"

### GIVEN/WHEN/THEN 模板

```markdown
### Scenario N: <场景名称（从用户视角描述）>

**Given** <用户当前状态/前置条件>
**And** <其他前置条件（可选）>
**When** <用户执行的具体动作>
**Then** <系统的可观察响应>
**And** <额外的可观察结果（可选）>
```

### 场景覆盖清单

每个功能至少包含：

1. **Happy Path**（主流程）
   - 正常使用场景，所有输入合法

2. **边界条件**
   - 最小/最大值
   - 空值/null 处理
   - 刚好达到限制的情况

3. **错误场景**
   - 非法输入
   - 权限不足
   - 资源不存在
   - 并发冲突

4. **业务规则**
   - 特定业务约束
   - 状态转换规则

### 验收条件编写规则

✅ 好的验收条件：
- AC-001：用户使用已注册邮箱登录，系统返回有效的访问令牌（格式：Bearer JWT）
- AC-P01：登录接口 P95 响应时间 < 200ms（并发100用户）

❌ 差的验收条件：
- AC-001：登录功能正常（太模糊）
- AC-P01：系统响应快（无法测量）

### 数据规格模板

```markdown
## 数据规格

### 输入 — <接口名称>

| 字段 | 类型 | 必填 | 约束 | 示例 |
|------|------|------|------|------|
| email | string | 是 | RFC 5322格式 | user@example.com |
| password | string | 是 | 8-128字符，含大小写+数字 | Passw0rd! |

### 输出 — 成功响应

| 字段 | 类型 | 说明 |
|------|------|------|
| accessToken | string | JWT，有效期15分钟 |
| refreshToken | string | JWT，有效期7天 |

### 错误响应

| 错误码 | HTTP状态 | 触发条件 |
|--------|---------|---------|
| AUTH_001 | 409 | 邮箱已注册 |
| AUTH_002 | 422 | 密码不符合强度要求 |
```

## Examples

### 示例：用户注册功能

```markdown
## Feature: 用户注册

### Scenario 1: 新用户成功注册

**Given** 我是一个未注册的用户
**When** 我提交邮箱 "newuser@example.com" 和符合要求的密码
**Then** 系统创建我的账号
**And** 系统发送验证邮件到我的邮箱
**And** 系统返回我的用户信息（含用户ID）

### Scenario 2: 邮箱已被注册

**Given** 邮箱 "existing@example.com" 已被其他用户注册
**When** 我尝试用 "existing@example.com" 注册
**Then** 系统拒绝注册
**And** 系统提示 "该邮箱已被使用"
**And** 不创建任何账号

### Scenario 3: 密码强度不足

**Given** 我尝试注册新账号
**When** 我提交密码 "123456"（少于8位且无大写字母）
**Then** 系统拒绝注册
**And** 系统提示密码强度要求
```
