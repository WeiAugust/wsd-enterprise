# WSD 深度访谈技能

## When to Use

- 用户描述的需求过于模糊（"做个XX功能"）
- 用户的需求可能有多种理解方式
- 需求涉及复杂业务规则时
- wsd-analyst 进行需求分析时

## How It Works

### 苏格拉底式提问框架

**第一轮：锚定范围**
```
核心问题：
1. 这个功能服务的是谁？在什么场景下使用？
2. 这个功能解决了什么具体问题？不做这个功能的代价是什么？
```

**第二轮：明确边界**
```
核心问题：
1. 成功实现后，用户能做到什么事情是现在做不到的？
2. 哪些情况明确不在这次需求范围内？
```

**第三轮：验收标准**
```
核心问题：
1. 如果我们明天就做完了，你会怎么测试它？举一个具体例子。
2. 什么情况下你会说"这个做得不对"或"不够好"？
```

**第四轮：约束和依赖**
```
核心问题（按需选择）：
1. 有什么技术约束或需要兼容的现有系统吗？
2. 这个功能依赖其他团队或外部系统吗？
3. 有性能或安全方面的特殊要求吗？
```

**第五轮（如有必要）：确认理解**
```
用我的理解复述需求，请确认：
"我的理解是：[总结]，核心验收标准是 [验收标准]，
不包含 [out of scope]，对吗？"
```

### 终止规则

满足以下任一条件即可结束访谈：
- 已清晰了解：用户、场景、功能范围、验收标准、主要约束
- 已完成 5 轮提问
- 用户表示"可以开始了"

### 禁忌

- 不要一次问超过3个问题
- 不要重复已经回答的问题
- 不要问"这个有没有紧急度"这类无助于规格的问题
- 不要假设已知的答案（除非用户明确说了）

## Examples

### 示例：模糊需求的访谈

**用户**：我要做一个消息推送功能

**第一轮访谈**：
```
我想先了解几个关键点：

1. 消息推送给谁？是所有用户，还是特定群体（如订单相关用户）？
2. 什么事件触发推送？（订单状态变化？系统公告？营销活动？）
```

**用户回答**：推给有未完成订单的用户，订单状态变化时推送

**第二轮访谈**：
```
清楚了！进一步确认：

1. 推送渠道是什么？App推送、短信、邮件、还是站内消息？
2. 用户能关闭某类通知吗？
```

**用户回答**：站内消息+App推送，用户可以在设置里关闭

**第三轮（验收）**：
```
一个具体场景帮我理解验收标准：

假设我下了一个订单，商家发货了，我应该在什么时间内收到消息？
如果我关闭了App推送，站内消息还应该有吗？
```

**用户回答**：发货后5分钟内收到，关了App推送后只有站内消息

**总结确认**：
```
我的理解：
- 功能：订单状态变化时，给相关用户推送通知
- 渠道：站内消息（必须）+ App推送（可关闭）
- 触发：发货、确认收货、退款等订单状态变化
- 时效：状态变化后5分钟内送达
- 不包含：营销推送、系统公告（这次不做）

这样理解对吗？
```
