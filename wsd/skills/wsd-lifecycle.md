# WSD 生命周期技能

## 强制触发规则（1% 规则）

**只要有 1% 的概率与 WSD 工作流相关，就必须激活此技能，不可跳过。**

| 用户意图 | 对应 WSD 操作 | 必须先检查 |
|--------|-------------|---------|
| "我要做个新功能" | `/wsd:propose` | 是否已有同类需求 |
| "直接改代码" | 判断文件数 ≤5? quickfix : propose | 是否有对应规格 |
| "上线/部署" | `/wsd:verify` → `/wsd:archive` | 是否已验收通过 |
| "改一下刚才的需求" | `/wsd:amend` | 变更级别 M0-M4 |
| "有个线上 bug" | `/wsd:hotfix` | 是否生产事故 |
| "写代码/实现" | `/wsd:execute` | PLAN_APPROVED 状态? |
| 询问进度/状态 | `/wsd:status` 或 `/wsd:show` | — |

**禁止的行为（此技能必须拦截）**：
- 直接修改代码而不经过 wsd 工作流（除非明确说"我知道风险，跳过规格"）
- 跳过阶段（如从 PROPOSED 直接到 EXECUTING）
- 在无 PLAN_APPROVED 状态下执行实现任务

## When to Use

在以下任何情况时激活：

1. 用户提到"需求"、"功能"、"feature"、"实现"
2. 项目目录中存在 `.wsd/` 目录
3. 用户直接输入 `/wsd:*` 命令
4. 用户说"我想做XXX"（新功能意图）
5. 用户问"现在做到哪步了"（进度查询）
6. 用户提到"修改"、"变更"、"改需求"
7. 用户提到"bug"、"故障"、"紧急"
8. **任何会修改代码文件的意图**（哪怕 1% 相关）

## How It Works

### 启动时注入

会话开始时，如果检测到 `.wsd/STATE.md` 存在：

1. 读取 STATE.md，展示活跃需求摘要
2. 如果有进行中的需求，提示下一步操作
3. 如果有阻塞的需求，高亮显示

**展示格式**：
```
📋 WSD 项目状态
  活跃需求：3 个
  进行中：REQ-20260414-001（EXECUTING，4/6任务）
  ⚠️ 阻塞：REQ-20260410-001（T-003 阻塞 48h）

继续实现：/wsd:execute REQ-20260414-001
查看看板：/wsd:status
```

### 阶段导航

用户表达新需求意图时：
```
用户："我想添加用户头像上传功能"

→ 引导到 /wsd:propose
→ 说明：这会启动深度访谈来明确需求范围
```

用户询问进度时：
```
用户："用户认证做完了吗"

→ 查找相关需求
→ 展示当前阶段和进度
→ 给出下一步操作
```

### 规格前置提醒

如果用户直接要求实现代码，但找不到对应规格：

```
⚠️ 未找到对应规格

"用户头像上传"没有对应的 WSD 规格文档。

建议先创建规格：
  /wsd:propose 实现用户头像上传功能

这样可以确保：
  ✅ 功能范围清晰，避免返工
  ✅ 有可验证的验收标准
  ✅ 实现过程有记录，便于团队协作

如果确实是紧急小修改，回复"跳过规格"继续。
```

## Examples

### 示例 1：新需求

```
用户：我要加一个黑暗模式切换

wsd技能：
📋 发现新功能需求

建议通过 WSD 工作流管理：
  /wsd:propose 添加黑暗模式切换功能

这会帮你：
  1. 明确功能范围（是全局切换？还是用户偏好设置？）
  2. 生成可验证的规格
  3. 拆解实现任务
  4. 跟踪实现进度
```

### 示例 2：进度查询

```
用户：wsd:status

wsd技能 → 展示状态看板（详见 wsd-status.md）
```

### 示例 3：阶段确认

```
用户：spec已经确认了，我们开始做计划吧

wsd技能：
检测到当前需求 REQ-20260414-001 状态为 SPEC_APPROVED

运行：/wsd:plan REQ-20260414-001
```
