# /wsd:quickfix

**用法**：`/wsd:quickfix <描述>`

**适用场景**：
- 文案/配置/样式调整
- 单个接口的小参数修改
- 简单 bug 修复（原因明确，修改范围清晰）
- 小功能增强（不影响现有接口契约）

**不适用**：
- 影响多个模块的改动
- 需要数据迁移的改动
- 改变现有接口 / 数据结构
- 需求本身还不清晰时

## 与 /wsd:propose 的区别

| | propose | quickfix |
|---|---------|---------|
| 规格 | 完整BDD规格 | 简化说明（1段话） |
| 技术方案 | 完整 design.md | 内联说明 |
| 任务拆解 | tasks.md | 直接执行 |
| 人工确认 | 每阶段确认 | 仅执行前一次确认 |
| 状态机 | 7阶段 | 3阶段 |

## 简化状态机

```
PROPOSED → QUICK_EXECUTING → QUICK_DONE
               ↓ 如果发现超出范围
            升级为完整需求
```

## 执行流程

### Step 1 — 评估变更范围

调用 `code-explorer` 快速分析改动范围：
- 影响文件数（超过5个 → 建议改用 `/wsd:propose`）
- 影响接口数（改变契约 → 必须改用 `/wsd:propose`）
- 存在数据迁移（→ 必须改用 `/wsd:propose`）

若范围超出，输出：
```
⚠️ 此改动范围超出 quickfix 适用场景：
  - 影响 8 个文件（建议阈值：≤5）
  - 涉及接口契约变更

建议改用完整流程：
  /wsd:propose <描述>

强制继续 quickfix：/wsd:quickfix --force <描述>
```

### Step 2 — 生成简化提案

不需要深度访谈，直接生成 3 项内容：
1. **改动说明**（1-3句话）
2. **影响文件列表**
3. **完成标准**（2-5条，可测试）

### Step 3 — 一次确认

展示简化提案，等待用户确认（单次，无多轮）：
```
⚡ Quickfix 提案

改动：修改用户列表接口的分页默认值从 10 改为 20
文件：src/users/users.controller.ts（1个文件）
完成标准：
  - [ ] 默认 pageSize 为 20
  - [ ] 原有传参方式不变
  - [ ] 单元测试更新通过

确认执行？(yes/no):
```

### Step 4 — 直接执行

确认后，直接调用 `wsd-executor` 执行，无需 plan 阶段。

### Step 5 — 自动完成

执行完成后：
- 创建需求记录（标记 `track: quickfix`）
- 记录审计日志
- 更新 STATE.md

## 升级为完整需求

执行过程中任何时候发现超出范围，可升级：
```
/wsd:upgrade <quickfix-req-id>
```
→ 将 quickfix 记录转换为完整需求，从当前状态继续
