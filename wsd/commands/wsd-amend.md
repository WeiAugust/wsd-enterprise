# /wsd:amend

**用法**：`/wsd:amend <req-id> <变更描述>`

**适用场景**：需求在任何阶段发生变更时使用。

## 变更分级与处理策略

```
wsd-analyst 分析变更影响 → 自动分级 → 决定回退深度
```

| 级别 | 条件 | 处理方式 |
|------|------|---------|
| **M0 文档级** | 仅描述/措辞调整，不影响实现 | 直接更新文档，继续当前阶段 |
| **M1 任务级** | 影响 ≤2 个已有任务，无新增接口 | 修改 tasks.md，重新执行受影响任务 |
| **M2 计划级** | 影响任务结构，但规格主体不变 | 回退到 PLAN_APPROVED，重新 plan |
| **M3 规格级** | 核心行为/接口契约改变 | 回退到 SPEC_APPROVED，重新 plan+execute |
| **M4 重大变更** | 需求目标根本改变 | 建议取消当前，重新 propose |

## 执行流程

### Step 1 — 影响分析（wsd-analyst）

分析变更描述，结合已有规格、任务和代码，输出：
- 变更级别（M0-M4）
- 受影响的规格场景（列举 Scenario ID）
- 受影响的任务（列举 Task ID）
- 已完成任务中需要返工的部分
- 预估额外工时

### Step 2 — 变更影响报告

```
📋 需求变更分析

变更：将用户列表接口改为支持多条件组合筛选
影响级别：M2（计划级）

影响范围：
  规格：Scenario 2（列表查询）需更新
  任务：T-003（查询接口）需修改，T-004（单元测试）需更新
  已完成任务：T-001、T-002 不受影响（可保留）
  新增任务：T-007（复合查询逻辑）

处理方案：
  1. 更新 functional.md Scenario 2
  2. 修改 tasks.md（T-003、T-004），新增 T-007
  3. 从 T-003 重新执行

预估额外工时：2.5h

确认变更？(yes/no):
```

### Step 3 — 执行变更

确认后：
1. 在 `meta.json` 记录变更历史（`amendments` 数组）
2. 更新受影响的规格/任务文档
3. 将状态流转到对应起点
4. 标记已完成但需返工的任务为 `NEEDS_REWORK`

## 变更记录格式（meta.json）

```json
{
  "amendments": [{
    "id": "AMD-001",
    "level": "M2",
    "description": "支持多条件组合筛选",
    "requestedBy": "weizhen",
    "requestedAt": "2026-04-15T10:00:00Z",
    "affectedSpecs": ["Scenario-2"],
    "affectedTasks": ["T-003", "T-004"],
    "newTasks": ["T-007"],
    "estimatedExtraHours": 2.5
  }]
}
```

## 特殊情况：执行中发现需求变动

如果在 `/wsd:execute` 过程中发现需求描述与实际业务不符，`wsd-executor` 会暂停并输出：

```
⚠️ 执行中发现需求歧义

任务 T-004 执行时发现：规格要求"按创建时间排序"
但产品原型图显示"按更新时间排序"

建议：暂停执行，澄清后继续
  /wsd:amend <req-id> 按更新时间排序（而非创建时间）
  /wsd:block <req-id> T-004 等待需求澄清
```
