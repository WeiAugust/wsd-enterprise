# /wsd:hotfix

**用法**：`/wsd:hotfix <描述> [--severity P0|P1]`

**适用场景**：生产环境故障、安全漏洞、数据问题，需立即修复。

## 热修复流程（≤30分钟到代码）

```
hotfix propose → 直接 execute（无 spec/plan） → verify → 归档 + 补文档
```

### Step 1 — 快速定位（5分钟）

调用 `wsd-analyst` 进行**极简访谈**（最多3个问题）：
1. 故障现象是什么？影响多少用户/请求？
2. 问题出在哪个模块/函数？（有错误日志吗？）
3. 预期修复方案是什么？

### Step 2 — 创建热修复记录

```json
{
  "reqId": "HOT-20260415-001",
  "title": "修复用户登录JWT验证空指针",
  "track": "hotfix",
  "severity": "P0",
  "status": "HOTFIX_EXECUTING",
  "startedAt": "2026-04-15T02:30:00Z",
  "targetResolutionAt": "2026-04-15T03:00:00Z"
}
```

热修复 ID 格式：`HOT-YYYYMMDD-NNN`（区别于正常 `REQ-`）

### Step 3 — 直接执行（wsd-executor）

跳过 spec 和 plan，直接进入 `wsd-executor`，附带：
- 故障描述
- 相关错误日志（如有）
- 定位到的代码位置
- 预期修复方案

**执行约束**：
- 只修改必要代码（最小化改动）
- 必须有回滚方案
- 修改后必须跑相关测试

### Step 4 — 快速验证

`wsd-verifier` 执行**简化验收**：
- 原故障场景是否已修复
- 相关测试是否通过
- 有无明显回归

### Step 5 — 上线后补文档（异步）

热修复合并后，24小时内补充：
```
/wsd:hotfix-followup HOT-20260415-001
```
→ 补充根因分析、防范措施、规格更新

## 安全约束

- P0 热修复可跳过 spec，但不可跳过**代码审查**（`wsd-reviewer` 强制执行）
- P1 热修复需至少一位 team-lead 确认才能合并
- 热修复记录永久保留，不可删除（合规要求）

## 与主线需求的关系

热修复独立于正常需求 ID 体系（`HOT-` vs `REQ-`），不影响正常需求看板。但热修复如果暴露了深层设计问题，应同时创建正常需求跟进：

```
/wsd:propose 基于HOT-20260415-001的根因，重构JWT验证模块
```
